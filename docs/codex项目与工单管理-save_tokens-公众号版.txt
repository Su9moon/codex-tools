codex的项目与工单管理：/save_tokens

一、我为什么要做这件事

这不是一个“先设计 Skill，再找场景”的项目，而是起源于一次被 Codex 反复折磨的经历。

当时我同时推进三个项目：一个项目刚推进一轮，另一个项目又要回头修订；边界没有定义清楚，任务没有稳定的结束点，做完之后还经常被打回重做。CC Switch 的七天统计显示：真实消耗 3,230,300,191 Tokens（约 32.30 亿），总请求数 24,346，总成本 $1,734.0321。这个数字不是成绩，而是一盏报警灯：我已经无法判断，消耗究竟来自项目本身、我的设计不清，还是 GPT 的执行偏差。

我真正想解决三个问题：区分项目开销；知道每个环节花了多少时间和 Token；分辨返工究竟来自需求、边界还是执行。只有留下事实，下一次才能更准确地估算成本、安排计划。

还有一个更隐蔽的问题：上下文窗口有限。对话越长，最初目标和已确认边界越容易离开当前轮次；多项目切换也会让我混淆任务。因此，项目记忆不能只放在聊天记录里。

二、Tokdash 解决了什么，又解决不了什么

我先询问 GPT 有没有能默认统计用量和工作习惯的工具。答案是：Tokdash 可以统计 Codex 会话的 Token、成本和时间。

于是我部署了 Tokdash。它能快速回答“这次会话用了多少”，却无法回答：哪个业务流程消耗最大？返工究竟来自我的方案，还是模型执行偏差？

会话是技术上的容器，却不是工程上的边界。要完成项目级、工单级的分析，必须让对话之外再有一层轻量的结构。于是，我开了一个新任务，开始编写自己的 Skill，并把它命名为 save-tokens——直白一点，就是“救救俺的 Tokens 吧”。

三、save-tokens 的设计

我没有把它做成复杂代理，而是建立“项目文档 + Skill + Git 记录”的轻量体系：项目文件保存上下文和任务，Tokdash 提供用量，Git 保存版本证据。

最初的 tickets 解决了编号和状态，却没有记录每个目标的消耗与返工原因。于是，tickets 负责索引，Skill 负责规则，Tokdash 负责事实。

最重要的是控制工单粒度：一个用户开发目标对应一个主工单，内部步骤、讨论和思考留在其中，不拆成无数小任务。（在这里我鉴定了60%以上的返工的确是我设计能力不足）目标固定为：分析、编写、讨论、修复、验证。任务记录优先级、状态、时间、用时、Token、Token events、成本、返工和差评。

快照机制负责形成闭环。建立工单时记录会话、开始 Token、成本和 Token events；执行完成后先汇报；只有用户明确认可，才能形成结束快照、计算差值并关闭工单（这里就是我对Gpt最残忍的报复。我要求他必须不顾一切争取我关闭工单）。缺失数据保持 unavailable，不估算、不补造。这样，“完成”不再是模型自己宣布的结果，而是可验证、可追溯的用户确认。

四、从 Skill 到 Tokdash 看板

基础流程完成后，我把它嵌入 Tokdash：复用已有会话索引，根据项目目录读取 TASKS.md 和任务详情。

实现过程从 codex-tools 基础、Tokdash 部署和 Skill 重命名开始，逐步增加项目关联、效率汇总、快照、Token events、筛选和手动刷新；每一轮都在真实项目中验证。

这一步经历了反复拉锯。工单一度拆得过细，导致快照不完整；页面的接入、筛选和关闭操作曾混在一起；刷新按钮看似工作，实际仍返回旧缓存；任务详情已完成，TASKS.md 却残留旧状态。每个问题都说明：自动化没有边界，只会把错误藏得更深（好专业啊你！）。

最后形成硬规则：接入状态只影响前端；手动刷新才清缓存；关闭工单同步详情、索引和报告并校验一致性；不一致直接报错。返工和差评必须明确记录。

五、最终结果

现在，魔改版 Tokdash 与 save-tokens 配合后，可以展示项目工单的用时、Token、Token events、成本、返工和差评，让我判断问题发生在目标、设计还是执行。

这也改变了我和 GPT 的协作方式：我会追问工单边界、开始快照、验证结果和返工原因，得到的是能延续到下一次对话的外部记忆。

按设计目标，Skill 只在建立任务、查询状态和收尾时读取必要信息，额外消耗理论上控制在整体 Token 的 1% 以内。它不承诺模型永远不犯错，而是让错误有记录、返工有原因、成本有依据。

这套方法的价值不在于多了几个字段，而在于把“聊天式开发”变成了“可恢复的工程过程”：先定义边界，再推进任务；先记录事实，再判断责任；用很少的额外消耗，换取长期可见的效率。
